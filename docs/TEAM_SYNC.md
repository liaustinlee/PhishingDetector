# PhishingDetector 改进同步说明

## 本次目标

这次优先完成两层提升：

1. 第一层：增强系统的工程鲁棒性，确保在模型不可用或 Key 失效时，不直接崩盘，而是自动切换到规则兜底模式。
2. 第二层：增强检测证据维度，重点补充邮件头校验（SPF / DKIM / DMARC）与附件风险等企业级安全信号，使系统的判定更像真实安全产品，而不是单纯 LLM 解释器。

---

## 第一层改进：无模型兜底能力

### 原始问题

此前项目的运行依赖真实 LLM API。只要 `MINIMAX_API_KEY` 为空、失效或网络异常，整个工作流会在第一个 Agent 就中断。

这会带来两个问题：

- 演示环境不稳定
- 评测环境一旦没有有效 Key，系统直接不可用

### 改进内容

我们在以下路径上补了稳态兜底逻辑：

- [src/llm.py](../src/llm.py)：将模型调用异常统一包装为可识别的 `LLMUnavailableError`
- [src/agents/semantic.py](../src/agents/semantic.py)：当 LLM 不可用时，自动切换到规则化语义判定逻辑
- [src/agents/detector.py](../src/agents/detector.py)：当 LLM 不可用时，启用规则增强的技术评估逻辑
- [src/agents/risk.py](../src/agents/risk.py)：风险阶段支持规则化兜底分数与等级输出
- [src/agents/response.py](../src/agents/response.py)：响应阶段支持处理动作的规则化兜底

### 结果

现在系统的行为变成：

- 模型可用：正常走 Agent + LLM 流程
- 模型不可用：自动退化为规则模式，不再整条链路崩掉

这使项目从“只能在最佳环境里跑”变成“在不理想环境下也能工作”。

## 第二层改进：增强安全证据维度

### 原始问题

项目原来主要依赖：

- 语义上文理解
- URL/发送者分析
- 关键词打标

这种方式能做“研究原型”，但距成熟安全产品仍然有明显差距：

- 缺少邮件头进行信誉/认证级别分析
- 缺少对附件风险的显式识别
- 缺少更强的企业级安全特征融合

### 改进内容

我们在检测阶段补上了更贴近真实安全产品的规则证据：

#### 1. 邮件头校验风险评估

从 `headers` 字段中读取 SPF / DKIM / DMARC 状态，并根据状态计算风险权重：

- `none` / `neutral`：增加一定风险分
- `fail`：显著提高风险

#### 2. 附件风险标记

如果邮件存在 `has_attachment = True`，则在检测结果里显式追加：

- `possible_attachment_scam`

#### 3. 内容标记增强

在规则链路中补充：

- `email_header_validation_failed`
- `suspicious_link`
- `credential_request`

这样做的作用是：

- 让最终判定不只看“文本语义”
- 让结果拥有更强的“安全证据链”
- 更接近企业邮件网关的风险分析思路

## 改进后的效果

### 现在的系统更像什么

相比原来的纯 LLM 解释型结构，现在的系统更接近：

- `LLM 解释层` + `规则兜底层` + `安全证据层`

这种架构的价值在于：

- 评测时更稳
- 演示时更强
- 失败时不掉线
- 输出更像安全分析过程，而不是单纯对话式回答
